# 자원 인지형 GPU 작업 스케쥴러 사용자 매뉴얼 (Single-User Resource-Aware Edition)

본 스케쥴러는 루트 권한($\text{sudo}$) 없이 작동하는 경량 GPU 작업 관리 시스템입니다. 사용자가 작업들을 큐에 등록하면 스케쥴러 데몬이 GPU의 VRAM 물리적 여유 용량과 작업별 예약량을 비교 평가하여 순차 및 병렬 실행을 제어하고, 실시간 모니터링을 통해 메모리 초과(OOM) 오류를 사전 차단합니다.

---

## 1. 사전 환경 설정 (Environment Setup)

스케쥴러의 모든 상태는 영구 파일 기반의 SQLite 데이터베이스(`gpusched.db`)에 기록됩니다.

### A. 환경 변수 등록
스케쥴러의 데이터베이스와 로그가 저장될 기본 디렉토리 경로를 설정합니다. 지정하지 않을 경우 기본값으로 사용자 홈 디렉토리 하위(`~/.gpusched`)에 생성됩니다.

*   **Linux/macOS (bash, zsh)**:
    ```bash
    export GPUSCHED_DIR="$HOME/.gpusched"
    ```
*   **Windows (PowerShell)**:
    ```powershell
    $env:GPUSCHED_DIR="C:\Users\user\.gpusched"
    ```

### B. 임계값(Threshold) 설정 (선택 사항)
시스템이 자원 가용 여부를 판별할 물리적 자원 한계 임계값을 지정할 수 있습니다.
*   `GPUSCHED_MEM_THRESHOLD`: 물리 GPU 사용 메모리 비율 상한 (기본값: `0.20` -> 20%)
*   `GPUSCHED_UTIL_THRESHOLD`: 물리 GPU core 사용률 상한 (기본값: `0.15` -> 15%)
*   `GPUSCHED_POLLING_INTERVAL`: 자원 스캔 및 큐 감시 주기 (기본값: `5.0`초)

## 2. 설치 및 실행 (Installation & Execution)

본 스케쥴러는 파이썬 패키지 설치 도구(`pip`)를 사용하여 로컬 환경에 손쉽게 설치하고 명령행 도구(`gpusched`)로 기동할 수 있습니다.

### A. 파이썬 패키지 설치
프로젝트 루트 디렉토리 (`gpu_scheduler`)에서 아래 명령어를 실행하여 설치를 진행합니다.
*   **개발자/편집 가능 모드 설치 (권장)**:
    ```bash
    pip install -e .
    ```
    소스 코드 변경 사항이 재설치 과정 없이 파이썬 실행 환경에 즉시 반영됩니다.
*   **일반 모드 설치**:
    ```bash
    pip install .
    ```

### B. 단축 명령행 도구 (`gpusched`) 실행
설치가 완료되면 `python -m gpu_scheduler.cli` 형태의 긴 명령어 대신, `gpusched` 라는 단축 명령어로 모든 스케쥴러 인터페이스를 호출할 수 있습니다.
```bash
# 데이터베이스 초기화
gpusched init

# 스케쥴러 데몬 백그라운드 시작
gpusched start

# 대기 큐 및 GPU 모니터 상태 조회
gpusched status
```
*   **PATH 환경 변수 설정 주의**: 윈도우 환경 등에서 설치 시 Scripts 경로가 `PATH`에 등록되어 있지 않다는 경고가 나타나는 경우, 출력된 경로(예: `C:\Users\user\AppData\Local\Packages\...\Scripts`)를 시스템 환경 변수 `PATH`에 추가해 주셔야 `gpusched` 명령어를 전역적으로 실행할 수 있습니다.

---

## 3. 초기화 및 마스터 데몬 구동

마스터 스케쥴러 데몬은 백그라운드에서 주기적으로 GPU의 가용 메모리 상태를 계산하고 큐에 적재된 작업을 직접 프로세스로 띄워 가동시킵니다.

### A. 데이터베이스 초기화
최초 실행 시 1회 수행하여 작업 저장 공간과 테이블 스키마를 초기화합니다.
```bash
gpusched init
```

### B. 스케쥴러 데몬 시작 (백그라운드 무인 가동)
작업 자동 실행 루프를 시작합니다.
```bash
gpusched start
```
*   실행 시 `gpusched.pid` 파일에 프로세스 아이디가 기록되며, 데몬의 동작 내역은 `gpusched.log`에 실시간으로 작성됩니다.
*   **주의**: 원격 터미널(SSH) 연결 해제 시 프로세스가 중단되는 것을 방지하기 위해, 반드시 `tmux` 가상 터미널 환경 내에서 시작하거나 `nohup` 명령어로 백그라운드 구동하는 것을 권장합니다.

## 4. 작업 등록 및 자원 요구량 제어 정책

작업을 큐에 등록할 때, GPU VRAM 자원 할당 및 예측은 다음 세 가지 정책에 따라 유기적으로 작동합니다.

### A. 명시적 VRAM 요구량 지정 등록
작업 실행 시 필요한 최대 VRAM 용량(단위: MB)을 사용자가 직접 입력할 수 있습니다.
```bash
gpusched add "python train.py --epochs 100" --req-mem 6000
```
*   스케쥴러는 대상 GPU의 VRAM 잔여 할당 가능 예산이 $6000\text{MB}$ 이상 확보될 때만 해당 작업을 기동합니다.

### B. 과거 이력 기반 자동 예측 등록 (VRAM 생략 시)
`--req-mem` 옵션을 생략하면 시스템이 작동 중 메모리 사용량을 스스로 학습합니다.
```bash
gpusched add "python train.py --epochs 100"
```
*   **과거 이력 있음**: 동일한 명령어 문패턴으로 성공 완료(`COMPLETED`)했던 과거 작업들의 최대 메모리 소모 기록($M_{\text{peak}}$)을 역산하여, $10\%$의 안전 마진을 더한 용량을 요구량으로 자동 지정합니다.
    $$M_{\text{req}} = \max \left( M_{\text{peak}} \right) \times 1.10$$
*   **과거 이력 없음**: 기본 설정 한계치인 `4000MB`를 요구량으로 임시 적용하여 기동시킵니다.

### C. 실시간 Watchdog 및 OOM 강제 회수 정책
작업이 실행(`RUNNING`) 상태로 진입하면, 데몬은 `nvidia-smi`를 통해 해당 자식 프로세스(PID)가 소모하는 실시간 VRAM 점유량 $M_{\text{active}}(t)$를 집중 모니터링하며 데이터베이스의 `PeakVRAM` 필드를 동적으로 갱신합니다.
*   **하드웨어 임계 위반**: 실시간 사용 메모리가 물리 GPU 전체 용량의 $95\%$를 초과하면 OOM 방지를 위해 프로세스를 즉각 강제 종료 시그널($\text{SIGTERM}$)로 자동 소멸시킵니다.
*   **예약 영역 초과**: 실시간 소모량이 사용자가 예약한 요구량($M_{\text{req}}$)을 초과하고, 물리 GPU 점유율이 위험 수위($90\%$)에 다다르면 프로세스를 자동 회수 처리합니다.

## 5. 실시간 상태 및 큐 모니터링

가동 중인 스케쥴러의 작동 상태 및 현재 큐의 VRAM 예약 분포 현황을 테이블 뷰로 확인합니다.

```bash
gpusched status
```

### 상태 모니터 출력 예시

```
====================================================================================
                        GPU Job Scheduler Status                                    
====================================================================================
Daemon Status : RUNNING
Daemon PID    : 14612
Storage Dir   : /home/user/.gpusched
------------------------------------------------------------------------------------
ID    Status     GPU   PID      ExitCode   ReqVRAM(MB)  PeakVRAM(MB) Command
----------------------------------------------------------------------------------------------------
1     COMPLETED  0     19960    0          4000         1500         python train.py --run 1
2     RUNNING    0     2304     -          1650         920          python train.py --run 2
3     PENDING    -     -        -          4000         0            python evaluate.py
====================================================================================
```

*   **Status (작업 상태)**:
    *   `PENDING`: 자원 부족으로 할당을 대기 중인 상태입니다.
    *   `ASSIGNED`: GPU가 배정되었으나 프로세스를 분기하기 직전의 과도 상태입니다.
    *   `RUNNING`: 스케쥴러에 의해 프로세스가 물리 기기 상에서 기동되어 연산이 돌고 있는 상태입니다.
    *   `COMPLETED`: 오류 없이 종료 코드 `0`으로 완료된 상태입니다.
    *   `FAILED`: 에러나 강제 취소, Watchdog OOM 세이프가드 작동으로 중단된 상태입니다.
*   **ReqVRAM(MB)**: 사용자 예약값 또는 과거 이력에 근거해 자동 예측된 VRAM 용량입니다.
*   **PeakVRAM(MB)**: 작업 실행 중 Watchdog이 계측한 실제 최대 VRAM 점유 피크값입니다. (성공 종료 시 다음 작업의 예측 기초 데이터로 활용됩니다.)

## 6. 실시간 로그 및 결과 확인

실행 중인 작업 혹은 완료된 작업의 통합 표준 출력(stdout) 및 표준 에러(stderr) 스트림은 로그 파일로 개별 보존됩니다.

```bash
# 특정 Job ID(예: 2번)의 실행 로그 확인
gpusched logs 2
```

---

## 7. 작업 취소 및 강제 중단

큐에 대기 중인 작업을 취소하거나, 현재 실행 중인 프로세스를 강제 종료하고 리소스를 즉시 회수할 수 있습니다.

```bash
# 특정 Job ID(예: 3번) 취소 및 kill 실행
gpusched kill 3
```
*   `PENDING` 또는 `ASSIGNED` 상태의 작업은 실행되지 않고 취소 처리됩니다.
*   `RUNNING` 상태의 작업은 프로세스 그룹 단위로 안전하게 동시 소멸 처리되며 데이터베이스 상태가 `FAILED` (종료 코드 `-15`)로 마감됩니다.

## 8. 데몬의 정상 종료 및 상태 복구

### A. 스케쥴러 데몬 정지
```bash
gpusched stop
```
*   활성화 상태였던 PID 파일이 소거되며 백그라운드 데몬 프로세스가 정상 다운됩니다. (가동 중이던 작업 프로세스는 영향 없이 독립적으로 보존되거나 종료 루틴에 따라 마무리됩니다.)

### B. 비상 상태 복구 및 휘발성 세션 보존
만약 서버 전원 공급 장해나 커널 크래시 등으로 인해 스케쥴러 데몬이 비정상 종료되어 프로세스가 고사하더라도 데이터베이스 파일은 안전하게 보존됩니다.
*   데몬을 새로 다시 기동(`gpusched start`)하면, 시스템이 데이터베이스를 스캔하여 이전 세션에서 비정상적으로 멈춘 작업(`ASSIGNED` 또는 `RUNNING` 상태로 남은 행렬)을 자동 감지합니다.
*   이후 모든 불완전 작업의 상태를 자동으로 `PENDING` 상태로 원복시키고 할당 정보를 초기화하여 큐의 첫 단부터 안전하게 다시 재실행 흐름을 타도록 구성되어 있습니다.
